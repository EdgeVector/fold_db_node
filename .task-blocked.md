# BLOCKED: `folddb transform test` + `transform test-suite` (TH2)

**Date:** 2026-04-29
**Branch:** `feat/transform-test-cli-th2`
**Task:** Phase 1 of transform test harness rollout, Wave 2 — ship `folddb transform test` and `folddb transform test-suite` CLI subcommands.

## BLOCKED reason

Two distinct dependencies are missing. CLAUDE.md and the task spec both forbid inlining raw HTTP in fold_db_node ("Use the existing schema_service_client crate ... Do NOT inline raw HTTP in fold_db_node"), so this work cannot proceed until typed wrappers exist in `schema_service_client`.

### 1. schema_service_client lacks `test_run` and `test_suite` typed wrappers

The current pin in `Cargo.toml` is:

```
schema_service_core   = { git = ..., rev = "f551a886b121ef579a5b30d4d26e679ea2884f85" }
schema_service_client = { git = ..., rev = "f551a886b121ef579a5b30d4d26e679ea2884f85" }
```

Verified at that rev:
- `crates/core/src/lib.rs` exposes `state`, `types`, `transform_resolver`, `trigger_simulate`, `wasm_validator`, etc. — but **no `transform_test_run` module and no `system_policy` module.**
- `crates/client/src/lib.rs` `SchemaServiceClient` impl exposes `add_schema`, `add_view`, `register_transform`, `simulate_trigger`, etc. — **no `test_run` or `test_suite` method, no policy fetch.**

### 2. TH1+TH5 endpoints exist on schema_service `main` but not in our pin

PR #87 in schema_service (`d2d22725f622ce65d89c1bee42cb099fa23411e4`) shipped:
- `crates/core/src/transform_test_run.rs` — `TestRunRequest`, `TestRunResponse`, `TestRunError`, `TestRunErrorKind`, `run_test()`, `MAX_INPUT_BYTES`, `DEFAULT_TEST_RUN_GAS`.
- `crates/core/src/system_policy.rs` — policy ETag + JSON serializer.
- `crates/server_shared/src/handlers.rs` — `test_run_transform`, `get_system_policy` actix handlers.

These are present at `main` HEAD, **not at our pinned `f551a88`.** The cross-repo bump-cascade bot will eventually pin a newer rev, but it has not yet (last bump cascade landed `f551a88` for fold_db rev `6293308a`).

Even when the cascade lands a newer rev, **the typed client wrappers (`SchemaServiceClient::test_run`, `::test_suite`) are not part of PR #87.** They have to be added in a follow-up to schema_service before this CLI work can use them.

### 3. TH3-server (`POST /v1/transforms/test-suite`) hasn't shipped at all

The task description states "TH3-server is in flight adding POST /v1/transforms/test-suite." Confirmed: no test-suite endpoint, no `TestSuiteRequest` / `TestSuiteResponse` types, no `run_test_suite` helper exists in any rev of schema_service today (verified by `git log --all --oneline | head -40` in the schema_service workspace — only TH1+TH5 landed in PR #87).

The canonical wire shape is fully pinned in `docs/designs/transform_test_harness.md` § "Test-suite endpoint shape (canonical)" so the CLI side knows the contract, but a client wrapper wrapping a non-existent server endpoint would be a speculative phantom dep.

## What unblocks this

1. **schema_service PR adding typed wrappers to `SchemaServiceClient`:**
   - `pub async fn test_run(&self, request: &TestRunRequest) -> FoldDbResult<TestRunResponse>` — POST `/v1/transforms/test-run`.
   - Re-export `TestRunRequest`, `TestRunResponse`, `TestRunError`, `TestRunErrorKind` from `schema_service_client` (or document `use schema_service_core::transform_test_run::*`).
   - Optional: `pub async fn get_system_policy(&self) -> FoldDbResult<PolicyDoc>` for TH5 (CLI may use it for skeleton command later).

2. **TH3-server lands on schema_service main** with `TestSuiteRequest` / `TestSuiteResponse` / `run_test_suite()` matching the canonical shape in the design doc, plus the actix + lambda handler.

3. **schema_service PR adding `SchemaServiceClient::test_suite`** — pairs with (2).

4. **bump-cascade to fold_db_node** picks up the new schema_service rev.

5. **Then** this CLI task can resume cleanly:
   - Add `Command::Transform { action: TransformCommand::{Test, TestSuite} }` to `src/bin/folddb/cli.rs`.
   - Wire dispatch in `src/bin/folddb/main.rs` — note these are NOT daemon-mediated (they hit schema_service directly via `SchemaServiceClient::new(&config.schema_service_url())`), so the dispatch path is closer to the upcoming `transform`-block than the existing `dispatch_http(...)` daemon-relay block.
   - New `src/bin/folddb/commands/transform.rs` with `test()` and `test_suite()`. Single-fixture mode: read `--input` JSON, build `TestRunRequest`, call `client.test_run()`. Batched mode: walk `--fixtures` dir for `*/{input.json, expected.json|expected_error.json}` pairs, chunk at 100, build `TestSuiteRequest`s, call `client.test_suite()`. Pretty-print summary lines + per-failure structured diffs; exit nonzero on any failure.
   - Integration test in `tests/transform_test_cli.rs` spinning a local schema_service test server (`schema_service_server_http` is already a dev-dep) and shelling out to the built `folddb` binary for both subcommands. The test-suite path needs TH3-server in place; until then, gate that test on a feature flag or skip-mark.
